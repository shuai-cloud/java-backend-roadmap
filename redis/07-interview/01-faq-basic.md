# Redis 基础面试题

---

## 1. Redis 是什么？有哪些特点？

**答案**：
Redis（Remote Dictionary Server）是一个基于内存的 key-value 结构非关系型数据库。特点包括：

- **纯内存操作**：读写速度极快，可达 10 万+ QPS。
- **单线程模型**：避免了锁竞争，所有命令原子执行（6.0 后引入多线程处理网络 IO，但命令执行仍是单线程）。
- **丰富的数据结构**：String、Hash、List、Set、ZSet、HyperLogLog、GEO、Stream 等。
- **持久化**：支持 RDB 和 AOF 两种持久化方式。
- **高可用**：主从复制 + Sentinel 哨兵实现自动故障转移。
- **分布式**：Redis Cluster 实现自动分片和高可用。
- **功能丰富**：支持事务、Lua 脚本、发布订阅、Pipeline、Stream 等。

---

## 2. Redis 的五大基本数据类型及使用场景？

| 类型 | 说明 | 使用场景 |
|------|------|----------|
| String | 字符串，最大 512MB | 缓存 JSON、计数器、分布式锁 |
| Hash | 键值对集合 | 存储对象（用户信息、商品详情） |
| List | 双向链表 | 消息队列、最新消息列表 |
| Set | 无序不可重复集合 | 标签、共同好友、抽奖 |
| ZSet | 有序集合（带 score） | 排行榜、延迟队列、限流 |

---

## 3. Redis 为什么快？

- **基于内存**：数据存储在内存中，读写速度远快于磁盘。
- **单线程模型**：避免了上下文切换和锁竞争，所有命令串行执行，保证原子性。
- **IO 多路复用**：使用 epoll（Linux）实现高效的网络 IO 处理。
- **数据结构优化**：针对不同数据类型使用高效的内部编码（如 ziplist、skiplist、hashtable）。
- **渐进式 rehash**：哈希表扩容时采用渐进式方式，避免一次性阻塞。

---

## 4. Redis 的过期键删除策略？

- **惰性删除**：当访问一个 key 时，检查是否过期，过期则删除。优点：节省 CPU；缺点：过期 key 可能长时间占用内存。
- **定期删除**：每隔一段时间（默认 100ms）随机抽取一部分设置了过期时间的 key，检查并删除过期 key。通过 `hz` 参数控制频率。
- **内存淘汰**：当内存达到 `maxmemory` 时，根据淘汰策略删除 key（如 allkeys-lru）。

实际是惰性删除 + 定期删除的组合策略。

---

## 5. Redis 内存淘汰策略有哪些？

| 策略 | 说明 |
|------|------|
| `noeviction` | 不淘汰，写操作返回错误（默认） |
| `allkeys-lru` | 从所有 key 中淘汰最近最少使用的（最常用） |
| `volatile-lru` | 从设置了过期时间的 key 中淘汰最近最少使用的 |
| `allkeys-lfu` | 从所有 key 中淘汰最不经常使用的（4.0+） |
| `volatile-lfu` | 从设置了过期时间的 key 中淘汰最不经常使用的 |
| `allkeys-random` | 从所有 key 中随机淘汰 |
| `volatile-random` | 从设置了过期时间的 key 中随机淘汰 |
| `volatile-ttl` | 从设置了过期时间的 key 中淘汰即将过期的 |

**推荐**：`allkeys-lru`。

---

## 6. 缓存穿透、缓存击穿、缓存雪崩的区别及解决方案？

| 问题 | 描述 | 解决方案 |
|------|------|----------|
| 缓存穿透 | 查询不存在的数据，每次穿透到 DB | 缓存空值、布隆过滤器、参数校验 |
| 缓存击穿 | 热点 key 过期瞬间，大量并发打到 DB | 互斥锁、逻辑过期（永不过期 + 异步更新） |
| 缓存雪崩 | 大量 key 同时过期或 Redis 宕机 | 均匀过期（随机 TTL）、多级缓存、Redis 高可用、限流降级 |

---

## 7. Redis 的持久化方式及对比？

| 特性 | RDB | AOF |
|------|-----|-----|
| 原理 | 内存快照（二进制） | 追加写操作日志 |
| 文件大小 | 小 | 大（可 rewrite） |
| 恢复速度 | 快 | 慢 |
| 数据安全性 | 可能丢失两次快照间的数据 | everysec 最多丢 1 秒数据 |
| 性能影响 | fork 子进程，主进程短暂阻塞 | 取决于 appendfsync 策略 |
| 适用场景 | 备份、快速恢复 | 数据安全性要求高 |

**推荐**：混合持久化（Redis 4.0+），AOF 重写时生成 RDB 格式 + 增量 AOF。

---

## 8. 什么是 Redis 事务？和关系型数据库事务的区别？

- **Redis 事务**：通过 `MULTI`、`EXEC`、`DISCARD`、`WATCH` 实现。
- **特点**：批量执行、不支持回滚（运行时错误不影响其他命令）、隔离性（事务执行期间不会被其他客户端命令打断）。
- **与关系型数据库的区别**：
    - Redis 事务不保证原子性（部分成功）。
    - Redis 事务不支持回滚。
    - Redis 事务通过 `WATCH` 实现乐观锁。
    - Redis 事务更轻量，性能更高。

---

## 9. Redis 的主从复制原理？

1. 从节点向主节点发送 `PSYNC` 命令。
2. 主节点执行 `BGSAVE` 生成 RDB 快照，同时将新写命令缓冲到复制积压缓冲区。
3. 主节点将 RDB 文件发送给从节点，从节点加载 RDB。
4. 主节点将缓冲区中的写命令发送给从节点，此后持续增量同步。
5. 从节点断开重连时，如果复制偏移量仍在缓冲区中，执行部分重同步；否则全量复制。

---

## 10. 什么是 Redis Sentinel？解决了什么问题？

- **Sentinel** 是 Redis 的高可用解决方案，用于监控主从架构，自动执行故障转移。
- **解决的问题**：
    - 主节点宕机时自动将从节点提升为主节点。
    - 客户端通过 Sentinel 获取当前主节点地址。
    - 提供通知功能。
- **部署要求**：至少 3 个 Sentinel 实例（奇数个），使用 Raft 算法选举领导者。

---

## 11. Redis Cluster 是如何分片的？

- Redis Cluster 将数据分为 **16384 个哈希槽**。
- 每个 key 通过 `CRC16(key) % 16384` 计算所属槽。
- 每个节点负责一部分槽（如 3 节点：0~5460、5461~10922、10923~16383）。
- 节点间通过 Gossip 协议交换元数据。
- 客户端直连任意节点，如果 key 不在当前节点，返回 MOVED 重定向。

---

## 12. 什么是缓存一致性？如何保证？

- **缓存一致性**：指缓存中的数据与数据库中的数据保持一致。
- **常用策略**：
    - **先更新数据库，再删除缓存**（推荐）：更新 DB → 删除缓存。下次查询时重建缓存。
    - **先更新数据库，再更新缓存**：写操作频繁时适用，但存在并发问题。
    - **延迟双删**：删除缓存 → 更新 DB → 延迟再次删除缓存（如 1 秒后）。
- **保证最终一致性**：使用消息队列异步重试删除缓存操作。

---

## 13. 什么是 BigKey？有什么危害？如何发现和处理？

- **BigKey**：占用内存过大或元素过多的 key（如 String > 10KB、集合元素 > 10000）。
- **危害**：阻塞操作、网络拥塞、数据倾斜、删除困难。
- **发现**：`redis-cli --bigkeys`、`MEMORY USAGE key`、定期扫描脚本。
- **处理**：拆分（如按时间分片）、压缩、设置 TTL 淘汰、使用 `UNLINK` 异步删除。

---

## 14. 什么是 HotKey？有什么危害？如何发现和处理？

- **HotKey**：短时间内被大量访问的 key。
- **危害**：单节点 CPU 飙升、集群数据倾斜、缓存击穿。
- **发现**：`redis-cli --hotkeys`（需 LFU 策略）、客户端统计、MONITOR（慎用）。
- **处理**：本地缓存（Caffeine）、副本分摊、打散为多个 key、限流降级。

---

## 15. Redis 的 Pipeline 是什么？有什么作用？

- **Pipeline**：将多个命令打包一次性发送给 Redis，然后一次性读取所有响应。
- **作用**：减少网络往返时间（RTT），显著提升批量操作的性能。
- **注意**：Pipeline 不保证原子性，命令可能被其他客户端插入。

---

## 16. Redis 的 Lua 脚本有什么优点？

- **原子性**：整个脚本作为一个整体执行，不会被其他命令插入。
- **减少网络开销**：一次发送，一次返回。
- **逻辑控制**：支持 if/else、循环等，实现复杂业务逻辑。
- **可缓存**：通过 `SCRIPT LOAD` 缓存脚本，后续用 `EVALSHA` 执行。

---

## 17. Redis 的发布订阅模式有什么缺点？

- 消息不持久化，订阅者离线丢失消息。
- 不支持消息确认和重试。
- 缓冲区满会断开订阅者连接。
- 适用于实时通知，不适合可靠消息传递。

---

## 18. Redis 的 Stream 相比 Pub/Sub 有什么优势？

- 消息持久化（存储在内存中，可配置）。
- 支持消费者组（负载均衡）。
- 支持消息确认（ACK）和重试（PEL）。
- 支持消息回溯（按 ID 范围读取）。
- 适合构建可靠的轻量级消息队列。

---

## 19. 什么是 Redis 的混合持久化？

- Redis 4.0+ 引入，同时开启 RDB 和 AOF 时，AOF 重写生成的文件包含 RDB 格式的快照 + 增量 AOF 日志。
- **优点**：兼具 RDB 的快速恢复和 AOF 的低数据丢失。
- **配置**：`aof-use-rdb-preamble yes`。

---

## 20. Redis 的单线程模型如何应对高并发？

- 单线程执行命令避免了锁竞争和上下文切换。
- 使用 **IO 多路复用**（epoll）高效处理大量客户端连接。
- 所有命令串行执行，天然原子性。
- 6.0 后引入多线程处理网络 IO 读写，但命令执行依然是单线程。

---

## 21. 如何保证 Redis 和数据库的双写一致性？

- **强一致性**：使用分布式事务（如 2PC）或锁，但性能较差。
- **最终一致性**：先更新数据库，再删除缓存。如果删除失败，通过消息队列异步重试。
- **延时双删**：先删缓存 → 更新 DB → 延迟再删缓存（如 1 秒），适用于读写分离场景。

---

## 22. 什么是 Redis 的渐进式 rehash？

- Redis 的哈希表在扩容时，不是一次性将所有元素 rehash 到新表，而是分多次逐步迁移。
- **过程**：维持两个哈希表（ht[0] 和 ht[1]），每次操作时顺便迁移一个桶，直到全部迁移完成。
- **优点**：避免一次性 rehash 导致的长时间阻塞。

---

## 23. Redis 的过期键对持久化有什么影响？

- **RDB**：生成快照时，已过期的 key 不会被保存到 RDB 文件。加载 RDB 时，也会过滤过期 key。
- **AOF**：AOF 文件不会包含过期 key 的删除命令。AOF 重写时，已过期的 key 不会被写入。

---

## 24. 如何实现 Redis 的分布式锁？有什么注意事项？

- **实现**：使用 `SET key value NX PX expireMs` 加锁，使用 Lua 脚本释放锁（确保只有持有者才能释放）。
- **注意事项**：
    - 设置超时时间，避免死锁。
    - 释放锁时校验 value（如 UUID），防止误删。
    - 考虑可重入性（Redisson 支持）。
    - 考虑自动续期（看门狗机制）。
    - 主从切换可能导致锁丢失（可使用 RedLock 算法）。

---

## 25. Redis 的 `KEYS` 命令有什么问题？替代方案是什么？

- **问题**：`KEYS pattern` 会遍历所有 key，时间复杂度 O(N)，会阻塞 Redis 主线程，导致无法处理其他请求。
- **替代方案**：使用 `SCAN` 命令增量迭代，每次返回少量 key，不会阻塞。

---

## 26. Redis 的 `INFO` 命令能查看哪些信息？

- `server`：服务器版本、运行时间、配置等。
- `clients`：已连接客户端数、最大客户端数等。
- `memory`：内存使用量、碎片率等。
- `persistence`：RDB 和 AOF 的状态。
- `stats`：命令执行次数、命中率等。
- `replication`：主从复制状态。
- `cpu`：CPU 使用情况。
- `keyspace`：各个数据库的 key 数量和过期 key 数量。

---

## 27. 如何提高 Redis 的命中率？

- 合理设置 TTL，避免过早过期。
- 缓存热点数据，冷数据走数据库。
- 使用本地缓存（如 Caffeine）作为一级缓存。
- 预热：在业务低峰期将热点数据加载到缓存。
- 避免缓存穿透、击穿、雪崩。

---

## 28. Redis 的 `SET` 命令的 `NX` 和 `XX` 选项有什么区别？

- `NX`：只在 key 不存在时设置成功（用于分布式锁）。
- `XX`：只在 key 存在时设置成功（用于更新已存在的 key）。

---

## 29. Redis 的 `EXPIRE` 命令的精度是多少？

- 精度为 **毫秒**。但过期时间以秒为单位设置，实际 Redis 会以毫秒精度检查过期。

---

## 30. Redis 的 `RENAME` 命令有什么风险？

- `RENAME` 会覆盖目标 key，如果目标 key 已存在，其值会被删除。建议使用前检查目标是否存在。
