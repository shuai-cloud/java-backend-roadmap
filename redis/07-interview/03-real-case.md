# 真实大厂面试场景分析

---

## 案例1：缓存穿透导致数据库被打爆

**背景**：某电商系统上线后，突然出现大量查询商品详情的请求，数据库 CPU 飙升到 100%，大量请求超时。

**排查过程**：
1. 查看 Redis 监控，发现缓存命中率从 95% 骤降到 20%。
2. 查看慢查询日志，发现大量 `SELECT * FROM product WHERE id = xxx` 查询，且这些 ID 在数据库中不存在。
3. 分析请求来源，发现是恶意攻击者遍历不存在的商品 ID。

**解决方案**：
1. 缓存空值：将不存在的商品 ID 也缓存起来，TTL 设为 60 秒。
2. 引入布隆过滤器：在查询前判断 ID 是否存在。
3. 参数校验：ID 必须为正整数，且不超过最大值。

**效果**：缓存命中率恢复到 92%，数据库负载恢复正常。

## 案例2：热点 key 击穿导致秒杀系统崩溃

**背景**：某秒杀活动开始时，商品详情页的缓存 key 刚好过期，数百万用户同时刷新页面，导致数据库连接池耗尽。

**排查过程**：
1. 查看 Redis 监控，发现该商品 key 的 QPS 高达 20 万。
2. 查看数据库监控，发现该商品的查询 QPS 瞬间飙升到 5 万。
3. 确认是该热点 key 过期导致的缓存击穿。

**解决方案**：
1. 使用互斥锁：只允许一个线程查询数据库并重建缓存，其他线程等待。
2. 逻辑过期：缓存永不过期，但在 value 中存储逻辑过期时间，异步更新缓存。
3. 本地缓存：在应用服务器本地缓存该热点数据（Caffeine），设置较短 TTL。

**效果**：数据库 QPS 降至 100 以内，秒杀系统平稳运行。

## 案例3：缓存雪崩导致全站瘫痪

**背景**：某社交平台在凌晨 0 点整，大量缓存同时过期，导致所有请求都打到数据库，数据库瞬间宕机。

**排查过程**：
1. 查看 Redis 监控，发现大量 key 在同一时间过期。
2. 查看代码，发现所有缓存都设置了相同的 TTL（24 小时），且都在 0 点开始缓存。
3. 确认是缓存雪崩。

**解决方案**：
1. 均匀过期：在 TTL 基础上增加随机值（±10%）。
2. 多级缓存：增加本地缓存（Caffeine）作为第一级。
3. Redis 高可用：使用 Sentinel 或 Cluster 避免单点故障。
4. 限流降级：对数据库访问进行限流，超出部分返回降级数据。

**效果**：数据库负载均匀分布，系统稳定运行。

## 案例4：BigKey 导致 Redis 阻塞

**背景**：某论坛的帖子详情页偶尔出现卡顿，Redis 的延迟从 1ms 飙升到 5 秒。

**排查过程**：
1. 使用 `redis-cli --bigkeys` 发现一个 ZSet 包含 500 万个元素。
2. 查看业务代码，发现该 ZSet 存储了全站的用户积分排行。
3. 每次查询排行榜时执行 `ZREVRANGE 0 -1`，导致 Redis 阻塞。

**解决方案**：
1. 拆分 ZSet：按积分区间分成多个 ZSet（如 0~1000、1000~5000 等）。
2. 限制查询范围：只返回 Top 100，使用 `ZREVRANGE 0 99`。
3. 使用 `ZSCAN` 代替全量查询。
4. 删除大 key 时使用 `UNLINK` 异步删除。

**效果**：Redis 延迟恢复到 1ms 以内。

## 案例5：分布式锁失效导致重复下单

**背景**：某外卖平台的订单系统出现少量重复订单，用户反馈被扣了两次钱。

**排查过程**：
1. 查看订单日志，发现同一用户在 10ms 内提交了两次订单。
2. 检查分布式锁实现，发现使用的是 `SETNX` + `EXPIRE` 两条命令，不是原子操作。
3. 在并发情况下，`SETNX` 成功后 `EXPIRE` 还未执行，锁就因异常而失效。

**解决方案**：
1. 使用 `SET key value NX PX expireMs` 原子命令。
2. 释放锁时使用 Lua 脚本保证原子性。
3. 引入 Redisson 分布式锁，自带看门狗自动续期。

**效果**：重复订单彻底消失。

## 案例6：主从切换导致缓存数据不一致

**背景**：某系统使用 Redis 主从架构，主节点宕机后 Sentinel 自动切换，但部分缓存数据丢失。

**排查过程**：
1. 查看 Redis 日志，发现主节点宕机时有部分写操作未同步到从节点。
2. 确认是异步复制导致的数据丢失。

**解决方案**：
1. 使用 `WAIT` 命令等待至少一个从节点确认（牺牲部分性能）。
2. 使用 RedLock 算法（但复杂度较高）。
3. 业务层面容忍短暂的不一致，通过定时任务补偿。
4. 使用 Redis Cluster 替代主从 + Sentinel（Cluster 的故障转移更安全）。

**效果**：数据丢失概率大幅降低，业务影响可接受。

## 案例7：Redis 内存耗尽导致服务不可用

**背景**：某系统的 Redis 实例突然无法写入，所有写操作返回 OOM 错误。

**排查过程**：
1. 查看 `INFO memory`，发现 `used_memory` 已达到 `maxmemory` 限制。
2. 查看淘汰策略，发现是 `noeviction`（不淘汰，写操作返回错误）。
3. 分析内存占用，发现大量缓存数据没有设置 TTL。

**解决方案**：
1. 将淘汰策略改为 `allkeys-lru`。
2. 为所有缓存数据设置合理的 TTL。
3. 增加 `maxmemory` 配置（从 4GB 扩大到 8GB）。
4. 使用 `redis-cli --bigkeys` 找出大 key 并拆分。

**效果**：Redis 恢复正常写入，内存使用率稳定在 70% 左右。

## 案例8：Redis 慢查询导致接口超时

**背景**：某管理后台的“数据统计”页面加载需要 30 秒，经常超时。

**排查过程**：
1. 查看 `SLOWLOG GET 10`，发现大量 `KEYS *` 命令，耗时 5~10 秒。
2. 追踪代码，发现统计功能使用 `KEYS` 遍历所有 key 来计算数量。
3. 该 Redis 实例有 500 万个 key，`KEYS *` 阻塞了主线程。

**解决方案**：
1. 将 `KEYS *` 替换为 `SCAN` 增量迭代。
2. 使用 `DBSIZE` 命令直接获取 key 总数（O(1)）。
3. 禁用 `KEYS` 命令：`rename-command KEYS ""`。

**效果**：统计页面加载时间从 30 秒降到 1 秒以内。

## 案例9：Redis 连接池耗尽导致应用无响应

**背景**：某微服务在高峰期出现大量连接超时，应用日志显示无法从连接池获取 Redis 连接。

**排查过程**：
1. 查看连接池监控，`maxActive` 为 50，但 `numActive` 始终为 50。
2. 查看代码，发现某处没有释放连接（忘记调用 `close()`）。
3. 确认是连接泄漏。

**解决方案**：
1. 修复代码中的连接泄漏问题。
2. 使用 try-with-resources 确保连接自动归还。
3. 增加连接池监控告警（`numActive` 接近 `maxActive` 时报警）。
4. 适当增大 `maxActive`（从 50 增加到 100）。

**效果**：连接池使用率恢复正常，无连接超时。

## 案例10：Redis 集群数据倾斜导致部分节点过热

**背景**：某 Redis Cluster 集群中，一个节点的 CPU 使用率高达 90%，而其他节点只有 20%。

**排查过程**：
1. 查看 `CLUSTER NODES`，确认各节点的槽分配均匀。
2. 使用 `redis-cli --hotkeys` 发现某个 key 的访问量占比 80%。
3. 确认是 HotKey 导致的数据倾斜。

**解决方案**：
1. 本地缓存：在应用服务器缓存该热点 key。
2. 副本分摊：为该槽添加只读副本，将读请求分散到副本。
3. 打散：将该 key 拆分为多个子 key（如 `hotkey:1` ~ `hotkey:10`），客户端随机访问。

**效果**：节点 CPU 使用率从 90% 降到 35%，集群负载均衡。
