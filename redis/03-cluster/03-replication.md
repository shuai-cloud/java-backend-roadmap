# Redis 主从复制

主从复制是 Redis 高可用的基础，它允许一个主节点（Master）的数据自动同步到一个或多个从节点（Replica/Slave）。从节点默认只读，可用于读写分离和备份。

---

## 一、复制的核心作用

- **数据冗余**：从节点保存主节点的数据副本，防止数据丢失。
- **读写分离**：主节点处理写请求，从节点处理读请求，提升吞吐量。
- **故障恢复**：当主节点宕机时，可以将从节点提升为新主节点（配合 Sentinel 或手动）。
- **数据备份**：可以在从节点上执行 `BGSAVE` 或 `AOF` 重写，不影响主节点性能。

---

## 二、复制的配置方式

### 1. 配置文件（从节点）
ini

指定主节点
replicaof 192.168.1.100 6379

如果主节点有密码
masterauth your_password

从节点是否只读（默认 yes）
replica-read-only yes

### 2. 运行时命令
bash

在从节点上执行
127.0.0.1:6380> SLAVEOF 192.168.1.100 6379

OK

取消复制（从节点变为主节点）
127.0.0.1:6380> SLAVEOF NO ONE

OK

## 三、复制的工作原理

### 完整流程（全量复制 + 增量复制）

1. **建立连接**：从节点向主节点发送 `PSYNC` 命令，携带主节点的 runid 和复制偏移量。
2. **全量复制**（首次连接或复制积压缓冲区中没有所需数据）：
    - 主节点执行 `BGSAVE` 生成 RDB 快照。
    - 主节点将 RDB 文件发送给从节点。
    - 从节点清空自身数据，加载 RDB 文件。
    - 在此期间主节点收到的写命令会缓冲在复制积压缓冲区中。
3. **增量复制**：RDB 加载完成后，主节点将缓冲区中的写命令发送给从节点，此后持续发送实时写命令。

### 复制积压缓冲区（Replication Backlog）
- 主节点维护一个固定大小的环形缓冲区（默认 1MB）。
- 记录最近的写命令，用于部分重同步（Partially Resynchronization）。
- 当从节点断开重连时，如果复制偏移量仍在缓冲区范围内，只需增量同步，避免全量复制。

### 心跳检测
- 主节点每隔 10 秒向从节点发送 `PING`。
- 从节点每隔 1 秒向主节点报告复制进度（`REPLCONF ACK {offset}`）。

## 四、复制的拓扑结构

### 1. 一主一从
最简单的结构，适合数据备份和读写分离。

### 2. 一主多从
主节点挂载多个从节点，分担读压力。但主节点需要将数据发送给所有从节点，可能增加主节点的网络和 CPU 负担。

### 3. 级联复制（链式复制）
Master → Slave1 → Slave2 → ...

Slave1 既是 Master 的从节点，又是 Slave2 的主节点。可以减少 Master 的直接从节点数量，适合从节点较多的场景。

## 五、复制相关的配置
ini

从节点只读
replica-read-only yes

复制积压缓冲区大小（字节）
repl-backlog-size 1mb

主节点向从节点发送心跳间隔（秒）
repl-ping-replica-period 10

复制超时时间（秒）
repl-timeout 60

是否禁用 TCP_NODELAY（yes 减少带宽，但可能增加延迟）
repl-disable-tcp-nodelay no

从节点优先级（Sentinel 选举时使用，越小越优先）
slave-priority 100

从节点是否允许写入（建议保持只读）
replica-read-only yes

## 六、复制的注意事项

### 1. 主节点内存
主节点在执行 `BGSAVE` 时会 fork 子进程，如果内存过大（> 10GB），fork 可能阻塞主进程数秒甚至更长。

### 2. 全量复制的网络开销
全量复制传输整个 RDB 文件，如果数据量大（几十 GB），可能耗尽带宽，影响正常业务。

### 3. 从节点数量
不建议主节点挂载过多从节点（建议 ≤ 3 个），否则主节点发送数据的压力过大。超过时可使用级联复制。

### 4. 数据不一致
主从复制是异步的，主节点写成功后立即返回，不等待从节点确认。因此主节点宕机时，未同步到从节点的数据会丢失。

### 5. 读写分离的延迟
如果从节点落后主节点较多，读到的可能是旧数据。可以通过 `WAIT` 命令（Redis 3.0+）强制等待从节点同步，但会影响性能。

## 八、常见问题

### Q1：主从复制延迟很大怎么办？
- 检查网络带宽和延迟。
- 增加 `repl-backlog-size` 以减少全量复制概率。
- 使用级联复制减少主节点直接连接的从节点数。
- 优化主节点的 `BGSAVE` 频率（避免频繁 fork）。

### Q2：主节点宕机后如何恢复？
- 如果启用了 Sentinel，会自动故障转移。
- 如果没有 Sentinel，手动选择一个数据最新的从节点，执行 `SLAVEOF NO ONE` 提升为主节点，然后修改其他从节点指向新主节点。

### Q3：从节点能否写入？
默认 `replica-read-only yes`，从节点不可写。如果强行写入，数据不会同步到主节点和其他从节点，且可能被后续全量复制覆盖。

---

## 小结

- 主从复制是 Redis 高可用的基石，实现数据冗余和读写分离。
- 全量复制 + 增量复制机制保证了数据同步的效率和可靠性。
- 注意复制是异步的，存在数据丢失风险。
- 配合 Sentinel 可实现自动故障转移。
