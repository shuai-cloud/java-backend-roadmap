## 一、面试高频问题 · 真人回答版

### 1. Redis 为什么快？
- 内存快，单线程省锁，IO 多路复用（epoll）。
- 数据结构设计好（ziplist、skiplist）。
- 渐进式 rehash 避免一次性阻塞。
- 缓存系统时间减少系统调用。

> 不要说太多细节，先抛主干，面试官追问再展开。

### 2. 为什么之前单线程，后来 6.0 又加多线程？
- 一直说的是**命令执行单线程**，6.0 只把**网络 IO 读写**改成多线程，命令执行还是单线程。
- 我们公司没开多线程，因为单线程够用，多线程调试麻烦。

### 3. 缓存穿透怎么解决？
- 首选**缓存空值**（简单有效）。
- 布隆过滤器维护成本高，有误判率，只在被攻击时考虑。
- 参数校验作为第一道防线。

### 4. 缓存雪崩怎么避免？
- 过期时间加随机值（必做）。
- Redis 高可用（Sentinel/Cluster）。
- 核心业务加本地缓存（Caffeine）。
- 数据库限流降级。

### 5. 分布式锁怎么设计？
- `SET key uuid NX PX 30000` 加锁。
- Lua 脚本释放锁（保证原子性）。
- Redisson 的 `RLock` 有看门狗自动续期。
- RedLock 太复杂，单节点锁 + 看门狗够用。

### 6. 缓存一致性：先更新 DB 还是先删缓存？
- 两种方案都有道理，没有绝对主流。
- **先更新 DB 再删缓存**：不一致窗口短，删除失败需 MQ 重试。
- **先删缓存再更新 DB**：需配合延迟双删，否则有脏读风险。
- 我们项目用第一种，因为对一致性要求高。

### 7. 热数据和冷数据？
- 热数据：频繁访问（首页、热门商品、登录会话）。
- 冷数据：很少访问（历史订单、日志）。
- 缓存只放热数据，冷数据放 DB 或归档。

### 8. Redis 哈希槽概念？
- 16384 个槽，CRC16(key) % 16384 决定归属。
- 每个节点负责一段槽，增删节点只需迁移槽。
- 客户端直连任意节点，返回 MOVED 重定向。

### 9. 过期策略？
- 惰性删除 + 定期删除（组合）。
- 内存淘汰策略（如 allkeys-lru）不属于过期策略，是内存不足时的兜底。

### 10. 集群什么情况不可用？写丢失？阻塞原因？
- **不可用**：超过半数 master 挂掉且无 slave 接替，或 `cluster-require-full-coverage yes` 且部分槽无节点。
- **写丢失**：主从异步复制丢数据，或网络分区后旧 master 的数据被丢弃。
- **阻塞原因**：慢命令（KEYS、SMEMBERS）、删除大 key、fork 阻塞、AOF fsync、内存 swap、CPU 争抢。

## 二、企业选型指南

### 1. 持久化选型
- 生产标配：**AOF everysec + 混合持久化**（`aof-use-rdb-preamble yes`）。
- 纯缓存可关持久化，但小心重启雪崩。

### 2. 事务 / Lua / Functions
- `MULTI/EXEC` 基本不用（伪原子性）。
- **Lua 脚本**是事实标准（真原子、一次网络往返）。
- Redis 7 Functions 是 Lua 的“正规军版”，但国内生产大多还在 6.x。

### 3. 消息队列选型
| 方案 | 场景 |
|------|------|
| List + BRPOP | 轻量任务队列，允许丢 |
| Pub/Sub | 实时广播，允许丢 |
| Stream | 可靠轻量 MQ，有 ACK 和消费者组 |
| 专业 MQ（RabbitMQ/Kafka） | 高吞吐、高可靠、延迟消息 |

### 4. 监控工具
- **本地排障**：RedisInsight（免费 GUI）。
- **生产监控 + 告警**：Prometheus + redis_exporter + Grafana。
- **云 Redis**：直接看云控制台（大 Key / 热 Key 分析）。

### 5. Java 客户端选型
- **新项目**：Spring Boot 默认 Lettuce（Netty 异步，线程安全）。
- **分布式锁/限流**：加 Redisson。
- **老项目**：Jedis + JedisPool 仍可用。

### 6. 哨兵 vs Cluster vs 代理
- **数据量 ≤ 单机内存 + 写不炸**：Sentinel（3 节点，quorum=2）。
- **数据超单机或写要水平扩展**：Cluster（3 主 3 从）。
- **代理**：基本淘汰，除非老系统客户端改不动。

## 三、常见误区与争议

### 1. “先删缓存再更新 DB” 是主流吗？
- 不是。两种方案都有，取决于场景。
- B站很多视频为了引出“延迟双删”才先讲这个方案，不代表生产主流。
- 面试时表现出你理解两种方案的 trade-off 即可。

### 2. RedLock 为什么被质疑？
- 依赖时钟同步，GC 停顿可能导致两客户端同时持锁。
- Redis 官方文档已移除 RedLock 推荐，Redisson 的 RedLock 已 `@Deprecated`。
- 单节点锁 + 看门狗对大多数场景够用。

### 3. BigKey 的阈值是死的吗？
- 不是。String 超过 10KB 或集合超过 5000 元素是参考值，还要结合访问频率。
- 一个 50KB 但每秒访问 1 次的 key 不是问题，一个 2KB 但每秒访问 10 万次的 key 才是热 key。

### 4. 提高缓存命中率就是加内存？
- 不完全是。更重要的是**预热、合理 TTL、LRU 淘汰、避免穿透**。
- 命中率不是越高越好，还要考虑数据新鲜度，一般 85%-95% 健康。

## 四、实战经验

### 1. BigKey 处理
- 发现：`redis-cli --bigkeys`（从节点跑），或 `MEMORY USAGE key`。
- 处理：拆分（按时间/业务分片）、压缩、`UNLINK` 异步删除。
- 预防：代码层面限制集合大小，超过阈值告警。

### 2. 阻塞排查
- 先看 `SLOWLOG`，再看 `LATENCY HISTORY`，最后看 `INFO` 的 `latest_fork_usec`。
- 禁用 `KEYS`、`SMEMBERS` 等危险命令。
- 大 key 用 `UNLINK` 代替 `DEL`。

### 3. 集群运维
- 槽迁移在低峰期做。
- `cluster-require-full-coverage no` 避免单槽故障导致全集群不可用。
- 监控每个节点的内存和 QPS，防止数据倾斜。

### 4. 缓存一致性实战
- 先更新 DB，再删缓存。
- 删除失败用 MQ 异步重试（3 次后告警）。
- 不要用延迟双删（增加接口延迟，且睡眠时间难确定）。
