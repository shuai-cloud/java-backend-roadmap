# Redis 持久化：RDB 与 AOF

Redis 虽然是内存数据库，但提供了两种持久化机制将数据保存到磁盘，以防止数据丢失。

---

## 一、为什么需要持久化？

- 防止进程退出或服务器宕机导致数据丢失。
- 用于数据备份、灾难恢复。
- 某些场景需要重启后快速恢复数据（如缓存预热）。

---

## 二、RDB（Redis DataBase）快照持久化

### 原理
RDB 是在指定的时间间隔内将内存中的数据生成一份**二进制快照**（dump.rdb），保存到磁盘。触发时，Redis fork 出一个子进程，子进程将数据写入临时文件，完成后替换原 RDB 文件。

### 触发方式
1. **手动触发**：
    - `SAVE`：阻塞 Redis 主进程直到 RDB 完成（不推荐生产使用）。
    - `BGSAVE`：fork 子进程异步执行（推荐）。
2. **自动触发**：
    - 配置 `save` 规则：`save 900 1`（900 秒内至少 1 次写操作）。
    - 从节点全量复制时，主节点自动执行 `BGSAVE`。
    - 执行 `FLUSHALL` 或 `SHUTDOWN` 时也会触发（如果开启了 RDB）。

### 配置示例：

ini

save 900 1

save 300 10

save 60 10000

dbfilename dump.rdb

dir /var/lib/redis

rdbcompression yes

rdbchecksum yes

stop-writes-on-bgsave-error yes

### 优点
- **文件紧凑**：二进制文件，体积小，适合备份和传输。
- **恢复速度快**：直接加载 RDB 文件到内存，比 AOF 快。
- **性能影响小**：fork 子进程处理，主进程几乎不受影响（但 fork 瞬间可能阻塞）。

### 缺点
- **数据安全性较低**：两次快照之间的数据可能丢失（例如每 5 分钟一次快照，宕机会丢 5 分钟数据）。
- **fork 阻塞**：如果数据集很大，fork 子进程会短暂阻塞主进程（毫秒到秒级）。
- **不适合实时持久化**：无法做到秒级以内丢失。

## 三、AOF（Append Only File）追加日志

### 原理
AOF 以**日志**的形式记录每次写操作（如 `SET`、`HSET`），Redis 重启时重新执行这些命令来恢复数据。AOF 文件格式，易于阅读和修复。

### 三种同步策略（appendfsync）
| 策略 | 描述 | 安全性 | 性能 |
|------|------|--------|------|
| `always` | 每次写操作都同步到磁盘 | 最高，最多丢失一个命令 | 最慢 |
| `everysec` | 每秒同步一次（默认） | 较高，最多丢失 1 秒数据 | 较快 |
| `no` | 由操作系统决定何时同步 | 最低，可能丢失大量数据 | 最快 |

### AOF 重写（Rewrite）
随着写操作增多，AOF 文件会越来越大。Redis 通过 **AOF 重写** 压缩文件：fork 子进程遍历内存中的数据，生成最小化的写命令集合（例如将多次 `INCR` 合并为一个 `SET`）。重写期间的新写操作会被缓冲，重写完成后合并。

#### 触发条件

ini

auto-aof-rewrite-percentage 100   # 当前 AOF 文件大小超过上次重写时的 100%

auto-aof-rewrite-min-size 64mb    # 最小触发大小

#### 手动触发

bash

redis-cli BGREWRITEAOF

### 优点
- **数据安全性高**：`everysec` 策略最多丢失 1 秒数据。
- **文件易读**，可手动修改修复（紧急情况下）。
- **支持 rewrite**：避免文件无限膨胀。

### 缺点
- **文件体积较大**：相同数据量，AOF 文件通常比 RDB 大。
- **恢复速度较慢**：需要逐条执行命令，比 RDB 慢。
- **可能有潜在 bug**：极端情况下 AOF 文件损坏导致无法加载（但 Redis 提供了 `redis-check-aof` 修复工具）。

## 四、混合持久化（Redis 4.0+）

### 原理
同时开启 RDB 和 AOF 时，AOF 重写不再生成纯 AOF 文件，而是生成 **RDB 格式的快照 + 增量 AOF 日志** 的混合体。这样兼具 RDB 的快速恢复和 AOF 的低数据丢失优势。

### 配置

ini

aof-use-rdb-preamble yes

### 加载顺序
Redis 启动时优先加载 AOF 文件（如果 AOF 开启），因为 AOF 数据更完整。如果 AOF 文件是混合格式，先加载内部的 RDB 部分快速恢复，再执行增量 AOF 命令。

---

## 五、持久化策略选择

| 场景 | 推荐方案 |
|------|----------|
| 纯缓存，允许丢失 | 关闭持久化（`save ""`、`appendonly no`） |
| 需要数据恢复，允许少量丢失 | RDB + AOF（everysec）混合 |
| 数据安全性极高（如金融） | AOF always + RDB 定期备份 |
| 磁盘空间有限 | 仅 RDB（体积小） |
| 需要快速重启 | RDB（加载快） |

## 六、常见问题

### Q1：RDB 和 AOF 可以同时开启吗？
可以。Redis 会优先使用 AOF 恢复数据（AOF 数据更完整）。如果 AOF 关闭，则使用 RDB。

### Q2：AOF 文件损坏怎么办？

bash

使用 Redis 自带的修复工具

redis-check-aof --fix appendonly.aof

### Q3：fork 子进程时主进程会阻塞吗？
fork 瞬间会阻塞主进程（拷贝页表），时间取决于内存大小。通常几毫秒到几十毫秒。建议控制实例内存不超过 10GB。

### Q4：BGSAVE 或 BGREWRITEAOF 期间发生写操作怎么办？
子进程共享父进程的内存快照（写时复制 COW），父进程的写操作会复制内存页，不影响子进程的快照。

---

## 小结

- RDB：快照方式，体积小、恢复快，但可能丢失较多数据。
- AOF：日志方式，数据安全性高，但文件大、恢复慢。
- 混合持久化：兼顾两者优点，推荐生产环境使用。
- 根据业务对数据安全的要求选择合适的策略。
