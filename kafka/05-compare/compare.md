# 消息中间件对比：Kafka vs RabbitMQ vs Redis Stream vs RocketMQ

> 选型是架构设计的重要环节。本文从多个维度对比主流消息中间件，帮助你根据业务场景做出合理选择。

---

## 一、总体定位

| 中间件 | 定位 | 典型场景 |
|--------|------|----------|
| **Kafka** | 分布式流处理平台 | 日志收集、大数据管道、流计算、事件溯源 |
| **RabbitMQ** | 传统消息队列 | 业务解耦、异步任务、RPC、延迟消息 |
| **Redis Stream** | 轻量级消息队列（Redis 模块） | 简单任务队列、实时通知、小规模消息 |
| **RocketMQ** | 阿里巴巴开源的分布式消息队列 | 电商交易、金融、IoT、大数据 |

---

## 二、详细对比

| 维度 | Kafka | RabbitMQ | Redis Stream | RocketMQ |
|------|-------|----------|--------------|----------|
| **吞吐量** | 极高（百万级/秒） | 较高（万级/秒） | 高（十万级/秒） | 极高（十万级/秒） |
| **消息持久化** | 磁盘，可配置保留时间 | 磁盘，可配置 | 内存+磁盘（RDB/AOF） | 磁盘 |
| **消息顺序** | 分区内有序 | 队列内有序 | 单 Stream 内有序 | 队列内有序 |
| **消费者模式** | 消费者组（点对点+发布订阅） | 多种模式（Direct/Topic/Fanout） | 消费者组 | 消费者组 |
| **消息确认** | Offset 提交 | ACK | ACK | ACK |
| **延迟消息** | 不支持原生（需时间轮） | 支持（死信+TTL） | 不支持原生 | 支持（定时消息） |
| **事务** | 支持（生产者事务） | 支持 | 不支持 | 支持 |
| **流处理** | 原生 Kafka Streams | 不支持 | 不支持 | 支持（轻量） |
| **运维复杂度** | 较高（需 ZooKeeper/KRaft） | 中等 | 低（依赖 Redis） | 较高 |
| **语言生态** | 多语言客户端 | 多语言客户端 | 多语言（通过 Redis） | 多语言 |
| **社区活跃度** | 非常活跃 | 活跃 | 活跃（Redis 生态） | 活跃（国内） |

## 三、选型建议

### 1. 选 Kafka 的场景
- 需要**超高吞吐**（日志收集、埋点、大数据管道）。
- 需要**消息持久化**且**回溯消费**（事件溯源、审计）。
- 需要**流处理**（实时计算、ETL）。
- 需要**多消费者组**独立消费同一份数据。

### 2. 选 RabbitMQ 的场景
- 需要**灵活的路由**（Direct/Topic/Fanout/Headers）。
- 需要**延迟消息**、**死信队列**。
- 需要**可靠的消息确认**和**重试机制**。
- 业务解耦、异步任务、RPC 调用。

### 3. 选 Redis Stream 的场景
- 项目已经使用 Redis，不想引入额外中间件。
- 消息量不大（十万级/天），对可靠性要求不高。
- 需要**轻量级**、**低延迟**的消息队列。
- 适合**实时通知**、**简单任务队列**。

### 4. 选 RocketMQ 的场景
- 国内互联网公司，需要**高可用**、**高吞吐**。
- 需要**事务消息**（如订单支付）。
- 需要**定时/延迟消息**（如订单超时取消）。
- 需要**消息轨迹**、**死信队列**等企业级特性。

## 四、面试常见问题

### Q1：Kafka 和 RabbitMQ 怎么选？
> 看场景。Kafka 适合大数据、日志、流处理，吞吐极高但运维复杂；RabbitMQ 适合业务解耦、异步任务，功能丰富且易用。如果只是简单业务消息，用 RabbitMQ；如果要做数据管道，用 Kafka。

### Q2：Redis Stream 能替代 Kafka 吗？
> 不能。Redis Stream 是轻量级方案，适合小规模场景。Kafka 在持久化、吞吐、流处理、多消费者组等方面远超 Redis Stream。Redis Stream 的优势在于简单，不需要额外部署。

### Q3：RocketMQ 和 Kafka 有什么区别？
> 两者都是高吞吐分布式消息队列。RocketMQ 对消息可靠性、事务、延迟消息的支持更好，更适合金融级场景；Kafka 在流处理和生态方面更强。国内很多公司用 RocketMQ 替代 Kafka。

---

## 小结

- 选型没有银弹，要根据业务场景、团队技术栈、运维能力综合考虑。
- 如果只是学习，推荐从 RabbitMQ 或 Redis Stream 入手，简单易用。
- 如果目标是进大厂，Kafka 是必学的，RocketMQ 在国内也很流行。
