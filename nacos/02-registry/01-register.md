# 服务注册与发现（进阶）

## 一、服务注册原理

Nacos 的服务注册基于 HTTP/gRPC 协议，流程如下：

1. 服务实例启动时，向 Nacos Server 发送注册请求（包含服务名、IP、端口、元数据等）。
2. Nacos Server 将实例信息存储在内存注册表中，并持久化到数据库（MySQL/Derby）。
3. 实例定期发送心跳（默认 5 秒）维持健康状态。
4. 消费者订阅服务时，Nacos 返回可用实例列表，并通过长轮询或 UDP 推送变更。

### 注册请求示例（Open API）

bash

curl -X POST 'http://localhost:8848/nacos/v1/ns/instance
' \

-d 'serviceName=order-service&ip=192.168.1.100&port=8080&weight=1&ephemeral=true'

---

## 二、服务发现原理

消费者通过 Nacos 获取服务实例列表，支持两种模式：

### 1. 客户端轮询（Pull）
- 消费者定时向 Nacos 拉取实例列表（默认 10 秒）。
- 优点：实现简单，不依赖长连接。
- 缺点：有延迟，无法实时感知变更。

### 2. UDP 推送（Push）
- Nacos 在实例列表变更时，通过 UDP 主动推送变更给订阅者。
- 优点：实时性强。
- 缺点：UDP 不可靠，可能丢包，客户端需配合轮询兜底。

### 3. gRPC 双向流（推荐，Nacos 2.0+）
- 客户端与 Nacos 建立 gRPC 长连接，服务端变更时直接通过流推送。
- 优点：实时、可靠，替代了 UDP 推送。

---

## 三、临时实例 vs 持久化实例

| 类型 | 特点 | 适用场景 |
|------|------|----------|
| **临时实例**（默认） | 心跳维持，心跳停止则自动剔除 | 微服务实例，动态扩缩容 |
| **持久化实例** | 手动注册/注销，不会自动剔除 | 固定服务（如数据库、Redis） |

通过 `ephemeral` 参数控制：

bash

curl -X POST '...&ephemeral=false'

---

## 四、健康检查

- **临时实例**：客户端主动心跳，Nacos 被动接收。若 15 秒无心跳标记为不健康，30 秒剔除。
- **持久化实例**：Nacos 主动探测（TCP/HTTP），探测失败则标记为不健康。

---

## 五、权重与保护阈值

### 权重
- 范围 0~1，默认 1。
- 权重越高，被消费者选中的概率越大。
- 可用于灰度发布（新版本权重设低）。

### 保护阈值
- 当健康实例比例低于阈值（如 0.5）时，Nacos 会将所有实例（包括不健康的）返回给消费者。
- 防止因大量实例不健康导致服务完全不可用（雪崩）。

---

## 六、优雅上下线

### 优雅下线
服务关闭前，主动向 Nacos 发送注销请求，消费者立即感知并移除该实例，避免请求打到已关闭的实例。

java

// Spring Boot 优雅停机（自动注销）

spring:

lifecycle:

timeout-per-shutdown-phase: 30s

### 优雅上线
- 服务启动后，延迟一段时间再注册，等待服务完全就绪。
- 可通过 `spring.cloud.nacos.discovery.metadata.startup` 标记状态。

---

## 小结

- 临时实例是微服务标配，心跳维持健康。
- 权重和保护阈值是服务治理的重要手段。
- 优雅上下线减少对业务的影响。
