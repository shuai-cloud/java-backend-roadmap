# 注册中心高可用

## 一、Nacos 集群部署

### 架构
客户端 → Nginx（负载均衡） → Nacos 节点1

→ Nacos 节点2

→ Nacos 节点3

↓

MySQL（共享存储）

### 部署要求
- 至少 3 个 Nacos 节点（奇数）。
- 所有节点连接同一个 MySQL 数据库。
- 前端加 Nginx 或 SLB 做负载均衡。

### 配置示例（cluster.conf）
bash

在 conf/cluster.conf 中列出所有节点 IP:Port
192.168.1.1:8848

192.168.1.2:8848

192.168.1.3:8848

### 启动集群
bash

sh startup.sh -m cluster

## 二、Nacos 节点间的数据一致性

- Nacos 使用 **Distro 协议**（自研）保证 AP 模式下的一致性。
- 写操作会同步到所有节点，但不需要等待所有节点确认（最终一致）。
- 如果切换为 CP 模式（`nacos.core.distro.type=raft`），则使用 Raft 协议保证强一致。

## 三、客户端高可用

- 客户端配置多个 Nacos 地址，自动切换。
  yaml

spring:

cloud:

nacos:

discovery:

server-addr: 192.168.1.1:8848,192.168.1.2:8848,192.168.1.3:8848

- 如果当前连接的 Nacos 节点宕机，客户端自动切换到下一个节点。

## 四、常见问题

### Q1：Nacos 集群中一个节点挂了，服务还能正常调用吗？
> 可以。只要还有节点存活，客户端会自动切换到其他节点。服务注册和发现不受影响。

### Q2：Nacos 集群中 MySQL 挂了会怎样？
> Nacos 节点本身还可以基于内存注册表提供服务，但重启后数据会丢失。所以 MySQL 也需要高可用（主从或集群）。

### Q3：Nacos 集群需要多少个节点？
> 至少 3 个。2 个节点在脑裂时无法达成一致，可能导致数据不一致。

## 小结

- Nacos 集群部署 + MySQL 共享存储是生产标配。
- Distro 协议保证 AP，Raft 协议保证 CP。
- 客户端配置多地址实现自动故障切换。
