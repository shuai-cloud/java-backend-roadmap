# Nacos 面试题精选（3-5 年经验）

> 以下题目覆盖 Nacos 的核心原理、实战经验和常见坑点。每个问题给出了“面试官想听什么”和“回答要点”。

---

## 一、基础概念

### 1. Nacos 是什么？解决了什么问题？
**回答要点**：Nacos 是阿里开源的服务发现和配置管理平台。解决了微服务架构中服务实例动态注册发现、配置集中管理和动态刷新的问题。相比 Eureka，它还内置了配置中心，功能更全。

### 2. Nacos 的核心组件有哪些？
**回答要点**：Nacos Server（集群）、Nacos Client（SDK）、服务注册表、配置存储（Derby/MySQL）、健康检查模块、长轮询机制、gRPC 通信。

### 3. Nacos 支持哪些服务发现协议？
**回答要点**：HTTP REST、gRPC（2.0+ 默认）。gRPC 比 HTTP 性能更好，支持双向流，心跳更轻量。

---

## 二、进阶原理

### 4. Nacos 的 AP 和 CP 模式有什么区别？怎么切换？
**回答要点**：
- AP（默认）：可用性优先，适合服务发现。使用 Distro 协议，最终一致性。
- CP：一致性优先，适合配置中心、分布式锁。使用 Raft 协议，强一致性。
- 切换方式：在配置文件或 API 中设置 `nacos.core.protocol.raft.mode`。

### 5. Nacos 如何实现配置的动态刷新？
**回答要点**：客户端通过长轮询（Long Polling）监听配置变更。当配置发生变化时，服务端立即返回响应，客户端更新本地缓存并触发 `@RefreshScope` 刷新 Bean。长轮询比定时轮询更实时，且节省服务端资源。

### 6. Nacos 的健康检查机制是怎样的？
**回答要点**：
- 客户端每 5 秒发送心跳（通过 gRPC 或 HTTP）。
- 服务端 15 秒未收到心跳，标记实例为不健康。
- 30 秒未收到心跳，剔除实例。
- 支持 TCP/HTTP 探测作为补充检查。

### 7. Nacos 的集群部署有什么要求？
**回答要点**：
- 至少 3 个节点（奇数，Raft 选举需要）。
- 必须使用 MySQL 作为外部存储（Derby 只适合单机）。
- 节点间通过 Raft 或 Distro 协议同步数据。
- 推荐配置 VIP 或 DNS 做负载均衡。

## 三、实战与排错

### 8. 服务注册后发现不了其他服务，可能是什么原因？
**回答要点**：
- Nacos 客户端和服务端网络不通。
- 命名空间或分组不匹配。
- 服务名拼写错误。
- 健康检查失败，实例被标记为不健康。
- 保护阈值设置过高，导致不健康实例也被过滤。

### 9. 配置修改后没有动态刷新，可能是什么原因？
**回答要点**：
- 类没有加 `@RefreshScope`。
- Data ID 或 Group 不匹配。
- 配置格式错误（如 YAML 缩进问题）。
- 客户端缓存未更新（长轮询超时时间未到）。
- 使用了 `@Value` 但没有加 `@RefreshScope`。

### 10. Nacos 集群脑裂怎么处理？
**回答要点**：Nacos 使用 Raft 协议保证 CP 模式下的强一致性，正常情况下不会脑裂。但如果网络分区导致多数节点失联，少数节点会拒绝写入，保证数据一致性。恢复网络后自动合并。AP 模式下不存在脑裂问题（允许不一致）。

### 11. Nacos 的性能瓶颈在哪里？如何优化？
**回答要点**：
- 服务端：注册表内存占用、长轮询连接数、gRPC 连接数。
- 优化：增加节点数、使用 MySQL 集群、调大长轮询超时时间、减少不必要的服务注册。
- 客户端：减少心跳频率、使用批量注册。

## 四、开放性问题

### 12. 你们公司为什么用 Nacos 而不是 Eureka？
**回答要点**：结合实际情况说明。例如：
- “我们用的是 Spring Cloud Alibaba 生态，Nacos 天然集成。”
- “我们需要配置中心，Nacos 二合一省一个组件。”
- “Eureka 2.0 已经停止开发，社区不活跃。”

### 13. 如果让你设计一个注册中心，你会怎么设计？
**回答要点**：可以参考 Nacos 的设计思路：
- 服务注册表：内存 + 持久化（数据库）。
- 健康检查：心跳 + 主动探测。
- 服务发现：客户端缓存 + 长轮询/推送。
- 高可用：集群 + Raft/Distro 协议。
- 可观测：控制台、指标、日志。

### 14. Nacos 和 Kubernetes 的服务发现有什么区别？
**回答要点**：
- K8s 通过 DNS 和 Endpoints 实现服务发现，对应用透明。
- Nacos 提供更丰富的功能（权重、保护阈值、健康检查细化）。
- K8s 适合容器化部署，Nacos 适合传统 VM 或混合部署。
- 两者可以共存：K8s 内部用 DNS，外部用 Nacos。

## 小结

- 面试 Nacos 时，重点理解 AP/CP 模式、长轮询机制、健康检查、集群部署。
- 结合实际项目经验（哪怕只是 demo）回答问题，会更有说服力。
- 准备好一个“你们公司用 Nacos 做什么”的故事，能串联起多个知识点。
